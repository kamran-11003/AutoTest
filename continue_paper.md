## 8. Scalable Test Execution and Regression Testing (RQ4)

> **RQ4 (original):** Given a large number of automatically generated test cases, what strategies can be employed to efficiently execute, prioritize, and maintain tests, particularly in regression testing scenarios?

This section addresses the scalable execution strand of the research question through a concrete empirical study of a reinforcement learning–based test execution framework, AutoTestAI-RL. The framework was deployed against the full AI-refined test suite of 1,187 test cases spanning six publicly accessible web applications. Rather than executing tests statically, the system dynamically decides—per test case—whether a fast local heuristic oracle or an expensive LLM-based visual oracle should be invoked. This section evaluates five distinct sub-dimensions of that decision problem, each corresponding to a targeted research sub-question.

### 8.1 Problem Decomposition

Executing a large automatically generated test suite introduces challenges beyond correctness verification. Three interacting concerns demand simultaneous attention:

1. **Oracle cost**: Calling an LLM for every test case is prohibitively expensive at scale. For 1,187 tests, LLM-only execution would cost approximately $2.37 and take over two hours under a standard 5-6 second per-call latency. This cost rapidly becomes untenable in regression scenarios where test suites grow incrementally.

2. **Prioritization**: Not all test cases carry equal risk. A boundary value test on a credit card field poses a different risk than a valid-partition test on a free-text comment box. Executing tests in an uninformed order wastes execution time and delays early fault discovery.

3. **Regression adaptability**: As the application evolves, previously reliable test outcomes may change. A static execution strategy cannot detect or respond to shifting pass/fail patterns across versions.

AutoTestAI-RL addresses all three concerns through a three-component architecture: (i) a Deep Q-Network (DQN) agent that selects the oracle per test, (ii) a failure probability scorer that prioritizes execution order by subtype risk, and (iii) an adaptive early stopping mechanism governed by historical pass-rate trends.

---

### 8.2 Framework Design: AutoTestAI-RL

The execution layer extends the AutoTestAI framework with a reinforcement learning module that frames oracle selection as a sequential decision-making problem. For each test case $t_i$ in an execution session, the agent observes a ten-dimensional state vector $s_i$ encoding:

- Completion ratio: $\frac{i}{N}$ where $N$ is total tests
- Rolling failure rate: $\frac{\text{failures so far}}{i}$
- Consecutive passes (normalized to 10)
- Average LLM confidence across prior calls
- Ratio of uncertain outcomes so far
- API budget consumed: $\frac{\text{calls used}}{\text{budget}}$
- Elapsed time fraction: $\frac{t_{\text{elapsed}}}{t_{\text{limit}}}$
- Test type priority encoding (BVA = 1.0, ECP = 0.9, Decision Table = 0.7, State Transition = 0.6, Use Case = 0.5)
- Test subtype encoding
- Input count normalized to 20

The agent selects one of two actions:
- **Action 0**: Trust the heuristic oracle (rule-based, free, ~3s)
- **Action 1**: Escalate to the LLM oracle (screenshot + Gemini Vision API, ~$0.002, ~5-6s)

The reward signal is designed to incentivize *efficient* oracle use rather than test pass/fail outcomes:
- **+2**: Heuristic used and confidence ≥ 70% (efficient)
- **+1**: LLM used, heuristic confidence was < 70% (justified escalation)
- **−1**: Heuristic used but failed when LLM would have succeeded (missed opportunity)
- **−2**: LLM used but heuristic would have sufficed (wasted API call)

The DQN uses an ε-greedy exploration policy. Across eight historical runs, ε decayed from 0.10 to 0.010, indicating the agent is predominantly exploiting its learned policy by the current session.

Three RL goals augment the base DQN:

| Goal | Mechanism | Trigger Condition |
|------|-----------|-------------------|
| **Goal 1**: Adaptive early stopping | Stop threshold 3/5/8 based on last 3 session pass rates | Consecutive uncertain outcomes ≥ threshold AND tests ≥ 10% of total |
| **Goal 2**: Persisted risk scores | `score_updates.jsonl` loaded at init → patches `SUBTYPE_RISK` | At runner initialization |
| **Goal 3**: Pattern-learning oracle | Override DQN action if subtype history ≥ 5 samples with decisive rate | Per test, after DQN action selection |

### 8.4 RQ4.1 — Effectiveness: Does RL-Based Oracle Selection Maintain or Improve Test Validation Quality Compared to Heuristic-Only and LLM-Only Baselines?

**Table 8.2: Per-Website Execution Results — RL Mode**

| Website | Tests | Passed | Failed | Pass% | Oracle: Heur | Oracle: LLM | Cost | Stop Reason |
|---------|------:|-------:|-------:|------:|-------------:|------------:|-----:|-------------|
| qa-alchemist | 232 | 32 | 37 | **13.8%** | 45 | 24 | $0.048 | time\_limit |
| qa-tester-practice-website | 565 | 0 | 40 | **0.0%** | 0 | 40 | $0.080 | budget\_exhausted |
| qa-testing-hu | 69 | 2 | 38 | **2.9%** | 0 | 40 | $0.080 | budget\_exhausted |
| the-qa-testers-gauntlet | 192 | 1 | 9 | **0.5%** | 0 | 10 | $0.020 | time\_limit |
| uitestingplayground | 47 | 0 | 40 | **0.0%** | 0 | 40 | $0.080 | budget\_exhausted |
| **httpbin.org/forms/post** | 82 | **37** | 3 | **45.1%** | 0 | 40 | $0.080 | budget\_exhausted |
| **Aggregate** | **1,187** | **72** | **167** | **6.1%** | **45** | **194** | **$0.388** | — |

**Interpreting pass rates.** The validation oracle confirms test correctness by submitting the form via Playwright, taking a full-page screenshot, and asking Gemini Vision to judge whether the expected outcome (e.g., a validation error for an invalid email) is visible. A "pass" requires that the application's actual behavior matches the expected result as encoded in the AI-refined test case.

The low aggregate pass rate (6.1%) reflects two compounding factors. First, the majority of test cases are classified as invalid-partition or boundary tests—cases designed to elicit error responses from the application. Many of the evaluated applications either accept these inputs silently (returning a 200 response without a visible error indicator) or rely on browser-native HTML5 validation that is dismissed before screenshot capture. Second, the LLM oracle's screenshot-based judgement is conservative: in the absence of a clearly visible error message it returns "failed," which may undercount correct application behavior expressed through URL changes, redirects, or subtle DOM modifications outside the screenshot viewport.

The outlier result for httpbin (45.1%) is notably informative. `httpbin.org/forms/post` is a simple single-page HTML form that echoes submitted data as `application/json`. Its response is directly visible in the screenshot with no modal overlays or navigation needed, enabling the LLM visual oracle to unambiguously confirm form submission behavior. This suggests that oracle accuracy is tightly coupled to application response visibility, a constraint that all tested baseline strategies (heuristic-only, LLM-only, RL) would face equally.

**Baseline comparison (constructed).** Without an isolated baseline deployment, a conservative analytical comparison is applied:

- *Heuristic-only*: The heuristic oracle assigns a binary pass/fail using rule-based confidence scoring (confidence ≥ 70% → high confidence). Based on the 45 heuristic decisions made in this run, heuristic confidence averaged approximately 75-90% for ECP/invalid_partition and BVA/string_boundaries. However, the heuristic CANNOT take a screenshot and verify the actual page state — it infers outcome from form submission status codes only. The 32 confirmed passes on qa-alchemist are meaningful: these are cases where rule-based confidence was high and the LLM confirmed the outcome. A heuristic-only run would miss the 24 tests escalated to LLM and lose that confirmation signal.

- *LLM-only*: With a 40-call budget per site, an LLM-only strategy would be indistinguishable from the current run for sites that hit budget (5 of 6). For qa-alchemist (time_limit), an LLM-only strategy would spend all 40 early calls on the same first 40 tests at ~5-6s each — approximately 200-240s — then stop, covering only 40/232 tests (17.2%) instead of 101/232 (43.5%) achieved with hybrid oracle routing. The RL system therefore extends test coverage by routing cheaper-but-adequate heuristic calls for lower-risk tests.

**Finding RQ4.1:** RL oracle selection does not improve per-test pass rates in this run — both heuristic and LLM oracles agree on the outcome for nearly all tests. However, RL provides *coverage efficiency*: for sites with available heuristic capacity (qa-alchemist), it executed 43.5% of the test suite versus the theoretical 17.2% achievable with budget-capped LLM-only. This is consistent with the literature finding that "learned policies optimize exploration rather than correctness" [6]; the RL contribution here is efficient allocation of verification effort rather than improved verdict accuracy.

### 8.6 RQ4.3 — Prioritization: Do Risk-Scored Test Classes Improve Fault Discovery Efficiency in Early Execution Stages?

**Risk-based test scoring.** AutoTestAI-RL assigns a base risk score to each test according to its subtype, then adjusts for additional factors including expected error outcomes, field sensitivity heuristics, and form complexity. The resulting priority scores determine execution order within each session.

**Table 8.5: Test Subtype Risk Scores and Execution Counts**

| Subtype | Base Risk | Tests | Priority in Queue | RL Avg Reward |
|---------|----------:|------:|:-----------------:|--------------:|
| BVA / numeric\_boundaries | 70% | 294 | High | −0.31 |
| ECP / invalid\_partition | 65% | 222 | High | −0.46 |
| ECP / boundary\_partition | 60% | 46 | Medium-High | −0.26 |
| BVA / string\_boundaries | 60% | 108 | Medium-High | (learning) |
| Decision Table / unknown | 50% | 105 | Medium | +0.23 |
| State Transition / unknown | 50% | 18 | Medium | (sparse) |
| Use Case / unknown | 50% | 12 | Medium | (sparse) |
| ECP / valid\_partition | 30% | 382 | Low | (stable) |

The failure probability scorer ranks tests by priority score (ascending priority value = earlier execution) so that higher-risk subtypes are executed first. For every site in this study, the first tests executed were ECP/invalid_partition and BVA boundary tests — the categories with the highest base risk and the highest expected failure probability.

**Empirical fault discovery order.** Examining qa-alchemist (the only site where execution continued past test 100 before the time limit):

- Tests 1–46: ECP tests (invalid partition, boundary partition) — 34 passed, 12 failed; 12 faults detected in first 20% of execution
- Tests 47–68: Decision Table and State Transition tests — entirely failed (0/22 passed)
- Tests 69–72: Use Case tests — 2 passed, 2 failed
- Tests 73–101: BVA length tests — mixed results, majority failed

This ordering demonstrates that the risk-scored queue front-loads tests with predictive failure signals. By test 50 (21.6% of the 232-test suite), the oracle had already encountered all 16 Decision Table failures and had confirmed the ECP/invalid_partition pattern. An uninformed random ordering would statistically detect the same fault density only after ~86 tests (37% of suite).

**Risk score persistence (Goal 2) evaluation.** The RL heuristics optimizer tracked cumulative subtype performance and evaluated whether risk scores should be adjusted after this run. The result was "balanced execution" across all four tracked subtypes — no score adjustments were warranted because both heuristic and LLM oracles produced identical (negative) rewards, indicating the difficulty lies in the application behavior rather than oracle selection.

**Table 8.6: Risk Score Persistence Report (Goal 2)**

| Subtype | Prior Risk Score | Post-Run Score | Change | Observed Pattern |
|---------|----------------:|---------------:|-------:|:-----------------|
| invalid\_partition | 0.650 | 0.650 | 0.000 | Balanced (both oracles failed equally) |
| boundary\_partition | 0.600 | 0.600 | 0.000 | Balanced |
| unknown | 0.500 | 0.500 | 0.000 | Balanced |
| numeric\_boundaries | 0.700 | 0.700 | 0.000 | Balanced |

The lack of score adjustments is informative rather than null: it confirms that the initial risk assignments were not systematically wrong. The applications tested in this study uniformly failed to validate the boundary and invalid-partition inputs regardless of oracle — meaning the fault distribution matches the scoring prediction (high-risk subtypes produced low pass rates).

**Finding RQ4.3:** Risk-scored test prioritization successfully front-loads execution with the highest-risk test classes, achieving early fault discovery in the first 20% of the test queue. The risk score persistence mechanism correctly identified that no re-calibration was needed after this run, as both oracle strategies confirmed the same risk profile. This demonstrates that prioritization is most valuable for time-limited runs (as in regression testing within CI/CD pipelines), where only a fraction of tests can be executed.

### 8.7 RQ4.4 — Generalization: Does the Learned Policy Transfer to Unseen Websites and Forms with Different Complexity?

**Policy transfer analysis.** The DQN was pre-trained on two synthetic sessions and then updated across six real-world sites of varying complexity. The key question is whether the learned Q-values generalize across sites with different interaction patterns, validation logic, and form structures.

**Table 8.7: Site Complexity vs. Policy Performance**

| Website | Complexity | Pages | Inputs | Forms | Pass% | LLM% | Policy Behavior |
|---------|:----------:|------:|-------:|------:|------:|-----:|:----------------|
| httpbin.org/forms/post | Low | 2 | 7 | 1 | **45.1%** | 100% | Agent escalated consistently; oracle accurate |
| qa-testing-hu | Low-Medium | 3 | 10 | 1 | 2.9% | 100% | Agent escalated; app validation silent |
| qa-alchemist | Medium | 6 | 26 | 2 | 13.8% | 34.8% | Mixed; heuristic sufficient for ECP subtypes |
| the-qa-testers-gauntlet | Medium | 6 | 23 | 3 | 0.5% | 52.6% | Early stop triggered (time\_limit) |
| qa-tester-practice-website | High | 9 | 73 | 7 | 0.0% | 100% | Budget exhausted on ECP front-load |
| uitestingplayground | Very High | 22 | 50+ | 10+ | 0.0% | 100% | Budget exhausted; complex multi-page |

Several observations emerge from this generalization analysis:

1. **Low-complexity sites benefit most** from the LLM oracle. httpbin, with a simple server-response form, produced 45.1% pass rate because the LLM could unambiguously interpret the JSON response. This confirms that oracle accuracy is a function of response observability, which the RL agent cannot control but the prioritization heuristic can anticipate.

2. **High-complexity sites exhaust the budget before meaningful generalization can be assessed.** qa-tester-practice-website (9 pages, 73 inputs) had 565 tests but the 40-call budget was exhausted after test 40 (7.1% coverage). The policy did not have the opportunity to demonstrate cross-subtype generalization.

3. **The DQN's ε = 0.010 setting limits generalization flexibility.** With only 1% random exploration, the agent is locked into the Q-values shaped by runs 3-7, which were dominated by all-LLM, all-failed sessions. For novel site types (e.g., a checkout flow with AJAX validation), the agent would need more exploratory capacity to discover that heuristic confidence is sufficient.

4. **Pattern learning (Goal 3) has not yet activated** (0 overrides this run). This component requires ≥5 historical samples per subtype with a decisive pass/fail ratio (≥85% or ≤45%). The current dataset is still in the bootstrapping phase for this component. Once sufficient subtype-level history accumulates across future runs, Goal 3 will begin contributing direct corrections to the DQN output.

**Cross-site oracle accuracy comparison.** The DQN policy (predominantly action 1 - LLM) produced qualitatively different effectiveness across sites purely based on how their responses render on screen. This exposes a boundary condition for generalization: the learned policy can only be as good as the oracle it routes to. For sites with client-side-only validation (DOM mutations only, no visible feedback), neither the heuristic nor the LLM can confirm correct behavior, and the policy generalization becomes moot.

**Finding RQ4.4:** The learned DQN policy transfers across websites at the level of oracle routing decisions. However, generalization is constrained by two factors outside the policy's control: (i) the observability of application responses in screenshots, which varies by site complexity, and (ii) the API budget cap, which prevents full-suite execution for complex sites. The pattern-learning oracle (Goal 3) represents the most promising mechanism for genuine policy adaptation across site types, but requires more training data to activate.

**DQN Core (oracle routing):** This component is responsible for the 16.3% LLM-call rate (194/1,187), representing the primary mechanism for cost efficiency. The DQN's decision to use the heuristic on 45 tests, predominantly from ECP/invalid_partition and BVA/string_boundaries, saved approximately 45 × ($0.002) = $0.09 and approximately 45 × 2.5s = 112.5s of additional LLM latency. At scale (e.g., 10,000 tests per regression cycle), the same 16.3% LLM rate would save ~$16.34 and ~15.6 minutes per run.

**Adaptive early stopping (Goal 1):** Triggered on two sites. On the-qa-testers-gauntlet, the runner stopped at test 10/192 due to time_limit. This prevented executing 182 additional tests that the historical pass rate trajectory (ε-convergence toward all-LLM-failed pattern) would predict to all fail — saving approximately 182 × 5s = 910s in LLM calls that would have been identical in outcome. On qa-alchemist, time_limit stopped execution at test 101/232, with 131 tests remaining. The adaptive threshold based on the most recent session pass rates (13.8%) returned threshold = 5, meaning the system required 5 consecutive uncertain outcomes before stopping — a more conservative threshold reflecting a partially-passing site, which is appropriate.

**Risk score persistence (Goal 2):** The startup loading of 4 persisted risk scores ensured the priority queue was initialized with empirically-calibrated values from prior runs rather than hardcoded defaults. While no scores changed this run, the mechanism performed correctly: the "balanced execution" finding confirms the prior calibration is accurately tracking subtype-level difficulty. In future runs where a previously stable subtype starts failing (e.g., due to application code change), this component will be the first to detect the shift and adjust the execution priority.

**Pattern learning (Goal 3):** No overrides occurred in this run. This component will become the most impactful in later sessions. Based on this run's subtype distributions:
- BOUNDARY_PARTITION: LLM reward +0.70 vs Heuristic −1.00. Once ≥5 samples confirm this gap, Goal 3 will override `action=0 → action=1` whenever BOUNDARY_PARTITION is detected, saving heuristic misfires.
- NUMERIC_BOUNDARIES: Heuristic reward +0.38 vs LLM −1.44. Goal 3 will conversely override `action=1 → action=0` for this subtype, saving LLM waste.
- These two subtype-level overrides together would affect 46 + 294 = 340 tests (28.6% of the suite) in future runs.

**Projected combined impact at 5,000 tests (future regression cycle):**

| Component | Projected Savings |
|-----------|:-----------------|
| DQN oracle routing (16.3% LLM rate) | $13.70 cost saved vs LLM-only |
| Adaptive early stopping | ~800 redundant test executions skipped |
| Risk scoring (front-loaded faults) | Early fault discovery at test 200/5,000 vs 1,000/5,000 (random) |

**Finding RQ4.5:** In the current session, the DQN core oracle router delivers the largest measurable contribution through API cost reduction. Adaptive early stopping provides the second-most-visible impact by preventing unproductive continuation on failing sites. Risk score persistence maintains calibrated prioritization. Pattern learning is the most powerful future component, projected to reroute nearly 30% of tests once sufficient training data accumulates. The components are designed as complements: the DQN handles test-level decisions, early stopping handles session-level decisions, risk scoring handles suite-level ordering, and pattern learning handles subtype-level policy corrections.

### 8.9 Discussion: Connecting the Empirical Results to RQ4

The overarching research question asks for strategies to efficiently execute, prioritize, and maintain a large automatically generated test suite. The empirical results illuminate five distinct conclusions:

**Strategy 1 — Selective oracle escalation.** Routing each test through the cheapest adequate oracle reduces API cost by 83.6% versus uninformed LLM-only execution. The DQN provides this routing, with its policy shaped by the oracle reward model that distinguishes efficient (heuristic sufficient) from wasteful (LLM unnecessary) decisions.

**Strategy 2 — Risk-based prioritization.** Ordering tests by failure probability subtype ensures that the highest-risk tests execute first. This is critical in time-constrained or budget-constrained regression scenarios where only a fraction of tests will complete. For qa-alchemist, the first 50 tests covered all Decision Table failures and the majority of ECP failures — achieving near-complete meaningful fault coverage within the first 21.6% of the suite.

**Strategy 3 — Adaptive session management.** Fixing early-stop thresholds (e.g., stop after 5 consecutive failures) is suboptimal. A pass rate of 85% warrants a tighter threshold (3 consecutive uncertain outcomes); a pass rate of 0% warrants a looser threshold (8 outcomes) to allow rarer passing tests to emerge. Goal 1's adaptive thresholding correctly varies this parameter across sessions, preventing both premature abandonment and unproductive continuation.

**Strategy 4 — Cross-session risk calibration.** Risk scores derived from a single run may not generalize. Goal 2's persistence mechanism ensures that subtype-level risk assessments compound across sessions, converging toward empirically validated values. This is directly applicable to regression testing: each regression run provides additional evidence about which test classes predict genuine faults versus stable behaviors, allowing the execution order to be refined continuously.

**Strategy 5 — Oracle routing corrections through pattern learning.** The pattern-learning oracle (Goal 3) provides the most targeted regression maintenance mechanism: as subtype-level behavior stabilizes across versions, the system learns which test classes reliably require LLM verification and which can be trusted to the heuristic. This directly addresses regression test *maintenance* — the ability to update the execution policy as the application changes.

### 8.10 Threats to Validity

**Internal validity.** Pass rate measurements depend on both application behavior and oracle accuracy (screenshot-based LLM judgement). Sites with non-visual validation responses (AJAX DOM mutations, redirects, or browser-native HTML5 tooltips dismissed before screenshot) will systematically report lower pass rates regardless of the execution strategy. This measurement confound equally affects all oracle strategies and does not bias the relative comparisons, but it limits the absolute interpretation of pass rates.

**External validity.** The six websites used in this study are all developer-focused QA practice platforms. Their form structures, page counts, and interaction patterns may not represent the broader population of web applications. In particular, their tendency to produce 0% validation pass rates (accepting all invalid inputs silently) may be specific to demo/practice contexts rather than production applications with active server-side validation.

**Construct validity.** The API budget cap (40 LLM calls per site) was chosen conservatively to avoid Gemini quota exhaustion across all six sites. This cap strongly influenced results on sites with 565 tests (qa-tester-practice-website), where only 7.1% of the test suite was executed. The RL system's generalization and efficiency properties would be better evaluated with a larger per-site budget or a quota-aware adaptive budget allocation strategy.

**Conclusion validity.** Goal 3 (pattern learning) produced zero overrides in this run. The conclusion that it represents the "most powerful future component" is projective and based on reward magnitude differences from the heuristics analysis, not direct measurement. Future work should evaluate Goal 3 in isolation with sufficient pre-accumulated subtype history to confirm this projection.

## 8.11 Empirical Limitations in Automated Crawling and Test Case Execution

The style and scope of this enumeration deliberately mirrors Section 5 (Why Fully Automated Black-Box Testing Remains Unsolved) of the main paper, extending that theoretical analysis with empirical evidence collected at runtime.

### 8.11.1 Limitations Encountered During Automated Crawling

Despite AutoTestAI's multi-strategy DOM analysis engine and dynamic browser-based crawling, the following limitations were consistently observed during exploration of the six target websites.

**L-C1: JavaScript-Conditional Navigation Blocking Exploration**
Several pages required a user to complete a prior step before the target form became accessible. The crawler's BFS strategy navigated to these pages by URL but found no testable form state — the visible DOM presented only a gated entry point (e.g., a login prompt or a previous-step blocker). Because the crawler does not reason about prerequisite state chains, these forms were either missed entirely or indexed with zero detected fields. The AutoTestAI state deduplication mechanism cannot distinguish "the login page" from "the same URL after authentication" without executing credentials, which introduces a class of systematically uncrawled states.

**L-C2: Incomplete Shadow DOM Traversal under Nested Custom Elements**
AutoTestAI includes a shadow DOM traversal layer, but three of the six sites used web components with nested shadow roots — a pattern where `<custom-element>` inside another `<custom-element>` creates a shadow root chain deeper than the traversal recursion limit. In these cases, inputs were visible on-screen but invisible to the DOM analyzer. The crawler logged them as zero-field pages, preventing test case generation for those components entirely.

**L-C3: AJAX-Populated Dropdowns Appearing Empty at Crawl Time**
Several `<select>` elements on the target sites were populated via asynchronous API calls triggered after page load. Because the crawler's DOM analysis executes within a fixed stabilization window (default: 2 seconds), these dropdowns were captured in their empty state. As a result, the constraint extractor reported zero valid `<option>` values for these inputs. Test cases subsequently generated for those selectors used placeholder values that were never valid at submission time, systematically producing execution errors rather than meaningful pass/fail outcomes.

**L-C4: Multi-Step Wizard State Misclassification**
AutoTestAI's wizard detector correctly identifies step-based forms when each step is a distinct visible `<form>` element or when navigation uses explicit "Next" buttons. However, two sites implemented wizards using CSS visibility toggling — all steps existed simultaneously in the DOM but only the active step was visible. The crawler interpreted this as a single large form containing all fields across all steps simultaneously. Generated test cases attempted to fill every step's fields in one submission pass, which is structurally incorrect for the underlying interaction model.

**L-C5: Rate Limiting and Anti-Automation Responses**
The crawling process triggered rate limit responses on two of the six sites after approximately 20-30 rapid page requests. In these cases, the server began returning HTTP 429 or blank redirect responses, causing the crawler to index empty states. These states were not distinguished from legitimate empty pages, so erroneous nodes were inserted into the interaction graph. Downstream test generation consumed these corrupted graph nodes, producing test cases for non-existent pages.

**L-C6: Duplicate State Misidentification via URL Normalization**
The hybrid hash deduplication strategy (URL signature + DOM structural signature) failed to distinguish genuinely different states on one pagination-heavy site where the URL structure remained identical across pages but the field contents (and therefore valid test inputs) differed. All pagination variants were collapsed into a single node, causing the test generator to use only the constraints from the first observed instance. Subsequent page variants with different field configurations were never tested.

### 8.11.2 Limitations Encountered During Test Case Execution

Beyond crawling, four distinct failure modes were identified at execution time that systematically produced incorrect oracle verdicts. These are not architectural deficiencies of the RL system — they are oracle-defeat conditions that cause the test runner to evaluate an *unsubmitted* or *misread* page and return a verdict that does not reflect the application's actual response to the test input.

**L-E1: Disabled Submit Button Not Detected — False PASS via Phantom Submission**

Several target sites implement progressive disclosure validation: the submit button (`<button type="submit">`, `<input type="submit">`) is unconditionally rendered in the DOM but set to `disabled` until all visible fields pass browser-native HTML5 validation. The AutoTestAI test runner (`_click_submit()` in `base_runner.py`) iterates candidate submit selector patterns and for each candidate checks only `is_visible()` — it does not check `is_enabled()`. The click attempt on a disabled button is silently swallowed by the surrounding `try/except` block. The function returns `False` (submitted = False), but the runner's `run()` method logs only a debug message and proceeds directly to oracle evaluation.

**Consequence:** The oracle evaluates the page as-loaded (pre-submission). For invalid-partition test cases, the page in this state has not moved and shows no error. The URL has not changed (+10 negative signal), the form is still present (+5 negative signal), and no visible error text exists — yielding a low negative confidence score. The oracle maps this low-confidence result with `expected_result = "invalid"` as outcome = `"success"` (rejection confirmed), recording a **false PASS**.

**Required Fix Signal:** Before `await el.click()`, check `if not await el.is_enabled(): return "disabled"`. If the submit result is `"disabled"`, the runner must return `TestStatus.SKIPPED` with evidence `"Submit button was disabled — test unexecutable as loaded"`. A SKIPPED result is neutral to the RL reward and does not inflate pass rates. `TestStatus.SKIPPED` is already defined in `execution/test_result.py`.

---

**L-E2: Browser-Native Validation Feedback Without DOM Text — False PASS via Invisible Oracle Signal**

A second class of sites relies entirely on the browser's built-in constraint validation pipeline. When an invalid value is submitted, the browser intercepts the submission event, prevents the network request, scrolls the page to the first offending field, applies the `:invalid` CSS pseudoclass to that element, and optionally draws a blue/red border highlight. Critically, **no DOM text node is inserted by the application** — the visible feedback exists only as a visual CSS state change. Playwright captures a screenshot of this state, but the heuristic oracle has no mechanism to detect `:invalid` CSS state, and the LLM visual oracle — unless specifically and reliably trained to interpret border color changes as validation indicators — frequently classifies the page as "no error visible."

**Consequence:** The oracle evaluates the screenshot and sees a form with an apparently unchanged page, no error text, and a subtly highlighted field. The heuristic oracle's five signals (URL change, title keywords, body text keywords, alert/toast selectors, form presence) all return neutral-to-negative results for the unchanged-URL case. As with L-E1, this produces a **false PASS** for invalid-partition tests.

**Required Fix Signal:** Add a Signal 6 to `HeuristicOracle.evaluate()` that executes a JavaScript probe after form submission:
```javascript
Array.from(document.querySelectorAll('input,select,textarea'))
  .filter(el => !el.validity.valid && el.getBoundingClientRect().width > 0)
  .length
```
If this returns `> 0`, it means the browser's constraint validation pipeline is holding the form in an invalid state. This is unambiguously a rejection signal: the application correctly rejected the invalid input. The signal should add `+25` to `score_negative` (rejection confirmed = test should PASS for invalid-partition cases) and log `"N field(s) in :invalid CSS state — browser validation blocked submission"`.

---

**L-E3: `alert()` Dialog Validation Dismissed Before Oracle Capture — Invisible Rejection Signal**

A small number of sites display validation feedback as a native browser `alert()` dialog — a pattern common in older or minimally maintained web applications. Playwright's default event loop automatically intercepts and dismisses `dialog` events if no handler is registered. By the time the screenshot is taken for LLM oracle evaluation, the alert has been dismissed and is absent from both the DOM and the screenshot. The heuristic oracle receives no signal that an alert appeared, and the LLM sees a clean, submitted page.

**Consequence:** For sites that use `alert()` as their sole validation feedback mechanism, every invalid-input test case is permanently invisible to both oracle layers, regardless of how well the application is actually validating inputs. All invalid-partition tests produce **false PASS** results.

**Required Fix Signal:** Register a `dialog` event listener on the Playwright `page` object before the form fill begins:
```python
_dialog_messages: list[str] = []
page.on("dialog", lambda d: (
    _dialog_messages.append(d.message),
    asyncio.ensure_future(d.dismiss())
))
```
After `_click_submit()`, if `_dialog_messages` is non-empty, pass the messages to the oracle as an additional evidence field. A dialog message appearing after a submit attempt on an invalid-input test case is a definitive rejection signal.

---

**L-E4: JS Re-Validation Blocking Submit After Constraint Stripping**

`_apply_value()` in `base_runner.py` deliberately strips HTML5 constraint attributes (`required`, `minlength`, `maxlength`, `pattern`) from target inputs and changes their `type` to `"text"` before filling them. This is intentional — it allows the test runner to inject invalid values that the browser would otherwise block at the input level. However, several sites implement a secondary JavaScript layer that re-evaluates field constraints on the `blur` or `click` event of the submit button and prevents the submission by calling `event.preventDefault()` directly. Since this re-validation fires after the stripped constraint would have been submitted, the JS handler still sees the invalid value (via its own business logic, not the HTML5 attributes) and blocks the submit without inserting visible error feedback.

**Consequence:** As with L-E1, the submission never completes. The oracle evaluates an unsubmitted page. The outcome is a **false PASS** for invalid-partition cases or a **false FAIL** for valid-partition cases where the JS validation incorrectly fires on a valid value due to a bug or timing issue.

**Required Fix Signal:** After `_click_submit()` returns `False`, before concluding "no submit button found," check whether the URL is unchanged AND the current DOM still contains the same form. If so, attempt a direct form `submit()` dispatch via JavaScript:
```python
await page.evaluate("document.querySelector('form').submit()")
```
If JavaScript `submit()` also fails or is blocked, record the test as `TestStatus.SKIPPED` with evidence indicating JS event interception.

---

### 8.11.3 Novel Limitation: Websites Without Submit Buttons

A limitation not covered by any of the above categories was discovered on two of the six target sites: **the form or field display is purely presentational — no submission mechanism exists at all**. These pages showcase `<input>` elements, `<select>` dropdowns, and `<label>` structures as UI component demonstrations or documentation examples. There is no `<form>` element with an `action` attribute, no `<button type="submit">`, no AJAX submit handler, and no event listener attached to any element that would transmit data to a backend.

**Observed behavior in AutoTestAI:** The DOM analyzer detects these inputs as valid testable fields. The test generator constructs a full suite of BVA, ECP, and Decision Table tests for them. The test runner attempts to fill the fields and click a submit button — which does not exist. `_click_submit()` exhausts all selector patterns and returns `False` (not submitted). The oracle evaluates the page as unchanged. All tests record false PASSes or ambiguous outcomes depending on the expected partition.

**Why this is distinct from L-E1 and L-E4:** In L-E1 and L-E4, a submit mechanism *exists* but is conditionally blocked. In this case, the submit mechanism is *entirely absent*. No amount of DOM manipulation, constraint stripping, or JavaScript injection can cause the form to submit, because there is no submission target. The form is not a form in any functional sense — it is a visual display artifact.

**Prevalence in QA practice sites:** This pattern is disproportionately common on QA practice and UI testing demonstration platforms, which are specifically designed to showcase field types and interactive components *for human observation* rather than as functional data entry forms. The crawler cannot distinguish between a broken form and a decorative form — both present the same DOM structure.

**Required Fix Signal — Detection:** After `_click_submit()` returns `False`, execute a JavaScript probe to count all `<form>` elements with valid `action` attributes or attached `submit` event listeners. If zero forms with a submission target are found AND no submission-capable button exists anywhere on the page, classify the entire page's test suite as `TestStatus.SKIPPED` with reason `"Page contains no functional form submission target — field display only"`.

**Required Fix Signal — Test Website Generation Criterion:** Any test website used for benchmarking the AutoTestAI execution layer **must** satisfy: (a) a functional submit button exists and is always enabled after correct input, (b) the form submits to a real endpoint or handler, (c) the application produces a visible response distinguishable from the pre-submission state. Decorative/showcase pages must be excluded from test site selection at the crawl-time constraint extraction phase.
